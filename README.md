# EngagePro Chatbot

**A RAG-powered customer service chatbot for EngagePro — routes company-specific queries through an internal knowledge base and general queries through Wikipedia, with a full modular backend pipeline and Streamlit UI.**

---

## Overview

EngagePro Chatbot is a retrieval-augmented generation (RAG) system built for EngagePro, a fictional AI solutions company. The chatbot answers questions about EngagePro's products and services using a private company knowledge base, and handles general or technical queries by searching Wikipedia — with a similarity-based router deciding which pipeline to use for every message.

Built with a locally deployed LLaMA 3.1 8B model via LM Studio, the system prioritises factual grounding, ethical safeguards and privacy by keeping all inference on-device.

---

## Architecture

The backend is a fully modular pipeline. Each file has a single, clearly defined responsibility.

```
engagepro-chatbot/
├── main.py                   # Entry point — launches Streamlit app
├── ui.py                     # Interface layer and session management
├── router_agent.py           # Central routing logic (document-first strategy)
├── rag_prepare.py            # Offline index builder — chunks + embeds brochure
├── rag_search.py             # Runtime vector similarity search
├── wiki_tools.py             # Wikipedia retrieval and content processing
├── clean_query.py            # LLM-based query reformulation agent
├── prompts.py                # Centralised prompt templates for all tasks
├── config.py                 # LLM configuration and connection settings
├── guardrail.py              # Output post-processing and safety checks
├── follow_up_questions.py    # Context-aware follow-up question generation
├── memory_manager.py         # Session persistence (user_memory.json)
├── conversation_summary.py   # Session summarisation agent
├── analytics.py              # Usage statistics (decoupled from answer pipeline)
└── data/
    ├── Company_Brochure.pdf  # EngagePro knowledge base source
    ├── vector_index/         # Pre-built embeddings (generated by rag_prepare.py)
    └── user_memory.json      # Persisted conversation history
```

### Request Flow

```
User message
    └── ui.py
        └── router_agent.py
              ├── [similarity ≥ 0.45] → rag_search.py → prompts.py → LLM → guardrail.py
              └── [similarity < 0.45] → clean_query.py → wiki_tools.py → prompts.py → LLM → guardrail.py
                                                                                    ↓
                                                                      follow_up_questions.py
                                                                      memory_manager.py
```

**Document-first routing:** every query is checked against the EngagePro knowledge base first. Only if the top similarity score falls below the threshold is the query redirected to Wikipedia. This prevents company questions from ever being answered with public sources.

## RAG Implementation

The knowledge base pipeline runs in two phases:

**Offline (rag_prepare.py):** The EngagePro company brochure is loaded, split into overlapping text chunks, and embedded into dense vectors using the model configuration defined in `config.py`. The resulting index is stored locally. Offline preparation avoids repeated embedding computation at runtime and ensures retrieval operates against a stable, reproducible index.

**Runtime (rag_search.py):** The incoming query is embedded using the same model configuration and a similarity search is run over the local index. The top-matching chunks and their scores are returned to `router_agent.py`. The score drives the routing decision; the chunks ground the final answer.

Company content and Wikipedia content are always handled through separate pipelines and are never mixed in the same response.

## Prompt Engineering

All prompt templates are centralised in `prompts.py`. Key techniques applied:

**Instruction-based prompting** — every prompt explicitly defines role, tone, scope of knowledge, and failure behaviour. The EngagePro prompt establishes the assistant as EngageBot and instructs it to respond from the company's perspective, never reveal internal tools or documents, and admit uncertainty clearly.

**Context-injection prompting** — retrieved text (from brochure or Wikipedia) is inserted directly into the prompt. The model is instructed to treat this as its own knowledge and is prohibited from referencing sources or documents in the response.

**Uncertainty and refusal prompting** — the model is explicitly required to say "I don't have that information" when retrieved context is insufficient, and speculative language is prohibited. This is the primary hallucination control mechanism.

**Zero-shot prompting** — used for answer generation, follow-up questions, and conversation summarisation. The model is guided by clear constraints rather than examples.

**Few-shot prompting** — used specifically for the query cleaning task. A small set of example input-to-query transformations is included to demonstrate the expected output format and behaviour.

**Output-constrained prompting** — supporting tasks receive strict format requirements: one cleaned query, one question per line (follow-ups), one sentence (summaries). This makes outputs directly usable downstream without post-processing.

Routing is handled by similarity score, not an LLM classifier — deterministic, faster, and easier to audit.

## Language Model

**Model:** `meta-llama-3.1-8b-instruct` via LM Studio (local deployment)

Selected for strong instruction-following in RAG settings and local inference, which keeps all user queries and conversation history on-device. The same model is used across all language tasks: answer generation, query reformulation, follow-up generation, and session summarisation. All model configuration is centralised in `config.py`.

---

## Safety and Ethical Safeguards

- **Guardrail layer** (`guardrail.py`) — post-processes every response to remove formatting artefacts, check for harmful content, and enforce conservative behaviour when retrieved context is weak
- **Refusal prompting** — the model is instructed to reject requests for illegal or harmful guidance and express uncertainty rather than generate unsupported information
- **Pipeline separation** — EngagePro and Wikipedia content are strictly isolated; company answers are never grounded in public sources
- **Privacy** — all inference is local; no user input or conversation history is sent to external services

## User Interface

Built with Streamlit. Key features:

- **Thinking indicator** — shown during retrieval and generation to prevent the interface appearing frozen
- **Source-specific icons** — different icons for EngagePro vs Wikipedia responses, making the knowledge source transparent without exposing routing details
- **Wikipedia source links** — displayed separately below Wikipedia-sourced responses for user verification
- **Clickable follow-up questions** — displayed after each response; clicking submits the question directly
- **Response length selector** — concise or detailed mode, passed to the backend and reflected in generation
- **Light / dark mode toggle**
- **Three tabs:** Chat (live interaction), History (past sessions with auto-generated summaries), Analytics (basic usage stats)
- **New Chat control** — resets session without losing history

## Running Locally

```bash
# 1. Clone the repo
git clone https://github.com/your-username/engagepro-chatbot.git
cd engagepro-chatbot

# 2. Install dependencies
pip install -r requirements.txt

# 3. Start LM Studio and load meta-llama-3.1-8b-instruct
#    Enable the local API server (default: http://localhost:1234)

# 4. Build the vector index (run once)
python rag_prepare.py

# 5. Launch the chatbot
python main.py
# or directly:
streamlit run ui.py
```

## Tech Stack

- **Frontend / UI:** Streamlit
- **Backend:** Python
- **LLM:** meta-llama-3.1-8b-instruct via LM Studio (local inference)
- **Embeddings & Vector Search:** sentence-transformers + FAISS (or equivalent, configured in config.py)
- **External Knowledge:** Wikipedia API
- **Storage:** JSON file (conversation history), local filesystem (vector index)
